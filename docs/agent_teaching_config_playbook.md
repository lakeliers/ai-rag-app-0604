# RAG Agent Pro 教学配置体验手册

## 1. 本轮验收结论

本轮目标是验证线上教学型 Agent 的配置按钮是否都能正常运行，并给出可体验差异的案例 prompt。

验收结果：

- 教学配置矩阵评测：33 个配置样本，33 个通过，通过率 100%。
- Smoke Set：9 个基础样本，9 个通过，通过率 100%。
- 本地 Streamlit 启动检查：首页返回 200 OK。
- LLM-as-Judge 已从 C 端聊天配置中移除，只保留在 Eval 评测体系中。

本地报告：

- `reports/teaching_config_matrix_report.html`
- `reports/teaching_config_matrix_report.json`
- `reports/teaching_config_smoke_auto_report.html`

注意：这次矩阵评测主要验证配置链路能正常跑通，不等同于评估每个策略在真实复杂场景下的答案质量优劣。策略质量仍需要 benchmark set 和真实样本评估。

## 2. 如何观察配置差异

不同配置项的差异不一定都直接体现在最终答案里。建议按三层观察：

1. 看最终回答：答案是否更完整、更贴近上传资料、是否引用网页资料。
2. 看参考来源：来源类型、来源顺序、chunk 类型、分数、向量排名、关键词排名、Rerank 状态。
3. 看 Agent 执行步骤：Planner 选择了什么动作，是否调用 web_collect、rag_search、generate_answer，Evaluator 是否拦截或放行。

建议把 Trace 展示级别调成“完整”，再测试下面的 prompt。

## 3. 资料来源策略

### 自动判断

体验目标：让 Agent 根据问题自己决定是否使用上传资料、网页资料或本地资料。

案例 prompt：

```text
RAG 是什么？
```

观察点：

- 通常会走本地知识库或已有资料。
- 如果没有强烈“最新”信号，一般不会强制联网。

### 仅上传资料

体验目标：验证 Agent 只使用当前上传资料，不引用历史上传文件或网页。

准备动作：上传一份与 Agent 学习相关的 PDF 或 MD。

案例 prompt：

```text
总结这份资料的核心观点
```

观察点：

- 参考来源里应主要是“上传资料”。
- 不应出现网页来源。
- 如果没有上传资料，应提示资料不足，而不是编造。

### 仅联网资料

体验目标：验证 Agent 只使用网页资料，不使用上传资料。

案例 prompt：

```text
最近 AI Agent 有什么新趋势？
```

观察点：

- 执行步骤里应出现 web_collect。
- 参考来源里应主要是“网络资料”。
- 即使左侧有上传文件，也不应优先引用上传文件。

### 上传资料 + 联网并行

体验目标：验证 Agent 同时使用上传资料和网页资料，然后做统一排序。

准备动作：上传一份 Agent 或 RAG 学习资料。

案例 prompt：

```text
结合我上传的资料，再查一下最近有没有类似案例
```

观察点：

- 执行步骤里应同时出现 web_collect 和 rag_search。
- 参考来源里应同时看到上传资料和网络资料。
- 上传资料通常用于理解你的上下文，网页资料用于补充最新案例。

## 4. 检索策略

### 仅向量检索

体验目标：观察语义相近但关键词不完全一致的问题能否被召回。

案例 prompt：

```text
不用术语解释一下，怎么让大模型先查资料再回答？
```

观察点：

- 更依赖语义相似度。
- 适合表达口语化、关键词不稳定的问题。
- 参考来源里关键词排名可能显示“未召回”。

### 向量 + BM25

体验目标：观察关键词命中对召回的帮助。

案例 prompt：

```text
BM25 和向量检索有什么区别？
```

观察点：

- 既看语义，也看关键词。
- 对包含明确术语的问题，通常比仅向量更稳定。
- 参考来源里应能看到关键词排名或 BM25 分数。

### 向量 + BM25 + RRF

体验目标：观察多路召回融合排序。

案例 prompt：

```text
RRF 在 RAG 检索里解决什么问题？
```

观察点：

- 同时看向量排名和关键词排名。
- RRF 会把不同召回链路的排名融合，降低单一路径误召回风险。
- 这是当前教学 Agent 的默认推荐检索策略。

## 5. Context Packing 策略

### 简单 TopK

体验目标：观察“只按排序取前 K 条”的效果。

案例 prompt：

```text
总结这份资料的三个核心观点
```

观察点：

- 最容易理解。
- 可能出现同一个文档连续多个 chunk 占满上下文。
- 适合作为对照组。

### 来源优先

体验目标：观察上传资料、网页资料、本地资料之间的来源配额。

案例 prompt：

```text
结合我上传的资料和网页资料，判断当前 RAG 产品化最应该补哪一块
```

观察点：

- 参考来源会更倾向保留不同来源类型。
- 不容易被某一个来源完全占满。

### 去重 + 新鲜度 + 来源权重

体验目标：观察综合分排序下，新资料和高优先级来源如何进入上下文。

案例 prompt：

```text
最近 Agent Memory 有什么实践趋势？结合已有学习资料给我建议
```

观察点：

- 新鲜度分数、来源权重会影响最终顺序。
- 上传资料和网页资料可能同时进入。

### 严格 token budget

体验目标：观察上下文预算控制，避免塞入过多资料。

案例 prompt：

```text
把这份长文档压缩成适合产品经理复习的提纲
```

观察点：

- 参考来源数量可能少于 TopK。
- 被跳过的资料可能显示超过上下文预算。
- 更接近生产环境里的上下文控制思路。

## 6. Chunking 策略

Chunking 策略只对新上传并入库的文件生效。已经入库的资料不会自动重新切分。测试时建议换文件名或重新上传新文件。

### 普通文本切分

体验目标：观察固定窗口切分。

建议上传内容：

```text
第一部分：RAG 的定义。
第二部分：RAG 的检索层。
第三部分：RAG 的评估方式。
```

案例 prompt：

```text
第二部分讲了什么？
```

观察点：

- chunk_type 可能是 plain。
- 实现简单，但可能切断段落语义。

### Parent-child

体验目标：观察子 chunk 命中后，保留 parent_text 作为上下文补充。

案例 prompt：

```text
这份资料里 RAG 检索层和评估层是什么关系？
```

观察点：

- chunk_type 通常是 child。
- 参考来源里可看到 parent 信息，适合长文档问答。

### 表格专用

体验目标：观察表头和行号是否被保留。

建议上传 CSV：

```text
姓名,状态,分数
A,通过,90
B,待确认,70
C,失败,40
```

案例 prompt：

```text
B 的状态是什么？
```

观察点：

- chunk_type 可能是 table。
- 参考来源里应保留表头、行号、sheet 等元数据。

### 摘要 chunk

体验目标：观察长文档是否生成摘要型 chunk，用于先召回主题，再带出原文依据。

建议上传较长文档。

案例 prompt：

```text
这份资料的核心结论是什么？
```

观察点：

- 可能出现 chunk_type 为 summary。
- 如果未配置可用模型或文本太短，摘要 chunk 可能不会生成。
- 摘要 chunk 适合大文档的主题级召回，但需要额外模型成本。

## 7. Planner 类型

### 规则 Planner

体验目标：观察稳定、可控的规则规划。

案例 prompt：

```text
你能做些什么？
```

观察点：

- 应直接回复能力介绍。
- 不应调用 web_collect 或 rag_search。

### LLM Tool Calling Planner

体验目标：观察由模型决定调用哪些工具。

案例 prompt：

```text
最近 AI Agent 有什么新趋势？
```

观察点：

- 理想情况下模型会选择 web_collect 和 rag_search。
- 如果模型规划失败，应回退到规则链路。
- 该模式更灵活，但稳定性、成本、延迟都更需要评估。

### fallback 混合 Planner

体验目标：观察当前默认教学链路：规则兜底 + 结构化 Runtime。

案例 prompt：

```text
结合我上传的资料，再查一下最近有没有类似案例
```

观察点：

- 既保留规则稳定性，又展示 Planner、Orchestrator、DAG Runtime、Aggregator、Evaluator 等阶段。
- 推荐作为默认教学模式。

## 8. Evaluator / Critic

### 关闭

体验目标：观察没有资料充分性判断时，Agent 直接进入生成。

案例 prompt：

```text
RAG 是什么？
```

观察点：

- Trace 里会显示 Evaluator 已关闭。
- 适合对照，但不推荐生产环境默认关闭。

### 规则评估

体验目标：观察低成本资料充分性判断。

案例 prompt：

```text
最近 AI Agent 有什么新趋势？
```

观察点：

- Evaluator 会判断资料数、来源类型、引用可用性。
- 这是 C 端聊天链路推荐保留的评估方式。

### 为什么没有 LLM-as-Judge

LLM-as-Judge 属于 Agent Eval 评测体系，不属于 C 端用户聊天体验配置。它应该在 smoke、regression、benchmark 测试报告里使用，而不是让普通用户在聊天页面开关。

## 9. Reranker

### 关闭

体验目标：观察不经过精排时，融合分排序的结果。

案例 prompt：

```text
Context Packing 是什么？
```

观察点：

- 参考来源里 Rerank 状态应为未启用。
- 速度更快，成本更低，但排序质量可能下降。

### 开启

体验目标：观察 reranker 对候选资料二次排序。

案例 prompt：

```text
Context Packing 是什么？
```

观察点：

- 参考来源里 Rerank 状态应为已启用。
- Rerank 分数会参与最终排序。
- 首次加载 reranker 可能较慢。

## 10. Trace 展示级别

### 隐藏

体验目标：模拟普通用户视角。

案例 prompt：

```text
你能做些什么？
```

观察点：

- 页面更简洁。
- 不适合调试。

### 简洁

体验目标：只看关键步骤摘要。

案例 prompt：

```text
RAG 是什么？
```

观察点：

- 能看到工具、状态和摘要。
- 适合教学演示。

### 完整

体验目标：查看每一步的原因、参数和结果摘要。

案例 prompt：

```text
结合我上传的资料，再查一下最近有没有类似案例
```

观察点：

- 能观察 Intent Classifier、Planner、Orchestrator、Runtime、Aggregator、Evaluator、Final Answer 的完整链路。

## 11. 推荐课堂演示顺序

1. 先用“你能做些什么？”验证直答路由，说明不是所有问题都应该 RAG。
2. 再用“RAG 是什么？”演示普通 RAG。
3. 上传一份资料，用“总结这份资料的核心观点”演示上传资料优先。
4. 用“最近 AI Agent 有什么新趋势？”演示联网 RAG。
5. 用“结合我上传的资料，再查一下最近有没有类似案例”演示上传 + 联网并行。
6. 切换检索策略，观察向量排名、关键词排名和 RRF。
7. 切换 Context Packing，观察参考来源数量、来源类型和上下文顺序。
8. 切换 Chunking 后重新上传文件，观察 chunk_type 和元数据。
9. 切换 Planner 类型，观察工具调用和回退行为。
10. 切换 Evaluator，观察资料评估阶段是否执行。

## 12. 当前限制

- 教学配置矩阵评测是 mock-plumbing 测试，主要保证配置不崩；真实质量需要 benchmark set。
- Chunking 只对新入库资料生效。
- 摘要 chunk 需要模型可用，且会带来额外成本。
- LLM Tool Calling Planner 的真实效果依赖模型工具调用能力和 prompt 稳定性。
- Reranker 首次加载可能较慢，线上环境可能受 Hugging Face 下载和限流影响。
